Sealion docs

Tailwind-like ergonomics without the Tailwind dependency.

Sealion components should use a compact utility-style grammar that the framework validates and compiles into deterministic CSS.

Product Decision

Sealion owns a small component styling system. Starter components live in .scale files under ui_components/l1, ui_components/l2, and ui_components/l3. Tailwind remains optional for apps that choose it.

Authoring Model

view/dashboard.skin
<s-l2.layout :passover=[
  title,
  app_name
]>
  <s-l3.dashboard-page :passover=[
    user_email
  ] />
</s-l2.layout>

ui_components/l3/dashboard_page.scale
<s-l2.page-header title="Dashboard" />

.skin views import .scale components and pass same-named variables with :passover=[...]. sealion format expands passover arrays into one variable per line. Explicit props remain available for aliases or literals, such as <s-l3.example :title="page_title" label="Save" />. Component composition is level-checked: skins may use L2/L3, L2 may use L1, L3 may use L1/L2, and L1 stays primitive. Dotted component names map to .scale paths.

Compilation Model

component source -> utility parser -> token resolver -> generated CSS artifact

The generated CSS is a build artifact. Component source and theme tokens are the source of truth.

Regression Requirements

  • Valid utilities compile and unknown utilities fail CI.
  • View files stay import-only and do not own CSS.
  • View and component calls pass required props explicitly.
  • Components use .scale, not .html.
  • Component calls obey the L1/L2/L3 hierarchy.
  • sealion format is idempotent for .skin and .scale files.
  • Generated CSS is deterministic.
  • Missing tokens produce actionable errors.
  • Components compile without Tailwind, Node, npm, or PostCSS.
  • Plain CSS remains an escape hatch, not the primary authoring path.
